Use case đề xuất

Hệ thống kiểm soát ra vào phòng lab bằng nhận diện khuôn mặt thời gian thực.

  • Admin tạo hồ sơ nhân sự hoặc sinh viên được phép vào khu vực.
  • Admin upload 3 đến 5 ảnh khuôn mặt mẫu cho mỗi người.
  • Worker nền sinh embedding và đồng bộ sang Qdrant.
  • Camera tại cửa vào gửi frame liên tục tới backend hoặc frontend gửi theo chu kỳ.
  • Backend gọi AI pipeline để phát hiện mặt và so khớp vector.
  • Hệ thống trả về danh tính, độ tin cậy và quyết định cho phép hay từ chối.
  • Log và ảnh snapshot được lưu để tra cứu sau.

Frontend người dùng

  • Hiển thị webcam hoặc luồng camera.
  • Gửi frame theo chu kỳ 1 đến 2 giây hoặc qua WebSocket.
  • Hiển thị ảnh, tên, confidence, trạng thái access.
  • Xử lý lỗi: mất camera, backend down, timeout.

Frontend admin

  • Đăng nhập quản trị.
  • CRUD người dùng và danh sách được phép truy cập.
  • Upload nhiều ảnh mẫu cho một người.
  • Xem log, xóa log, cấu hình threshold.

Backend API

  • Quản lý auth, route, validation dữ liệu.
  • Nhận frame, gọi AI service, ghi log.
  • Lưu metadata ảnh, đường dẫn MinIO, cấu hình hệ thống.
  • Tạo job nền sang worker.

AI service

  • Face detection và face alignment.
  • Trích xuất embedding từ ảnh.
  • So khớp với vector database.
  • Áp dụng threshold để quyết định known hay unknown.

Worker + Queue

  • Sinh embedding sau khi upload ảnh mới.
  • Re-index lại vector khi xóa hoặc cập nhật người dùng.
  • Chạy tác vụ nền để user không phải chờ.
  • Lưu trạng thái job để admin theo dõi.

Lưu trữ và proxy

  • PostgreSQL lưu user, logs, config.
  • MinIO lưu ảnh gốc và snapshot.
  • Qdrant lưu embedding khuôn mặt.
  • nginx điều phối request tới frontend và backend.

Quy trình 7 bước

  1. Admin tạo hồ sơ người dùng mới và upload 3 đến 5 ảnh khuôn mặt.
  2. Backend lưu metadata vào PostgreSQL và file ảnh vào MinIO.
  3. Backend tạo một job mới, đẩy vào Redis hoặc RabbitMQ.
  4. Worker nhận job, tải ảnh từ MinIO và thực hiện detect hoặc align.
  5. Worker sinh embedding cho từng ảnh, gắn với person_id.
  6. Worker upsert vector vào Qdrant để phục vụ truy vấn gần nhất.
  7. Worker cập nhật trạng thái job để admin biết ảnh đã được index xong.

Ý nghĩa kỹ thuật

Việc tách bước sinh embedding ra background worker giúp frontend admin không phải chờ lâu, đồng thời thể hiện đúng yêu cầu đề bài về message queue và tác vụ nền.

Luồng nghiệp vụ chính

  1. Camera hoặc trình duyệt lấy frame theo chu kỳ cố định.
  2. Frontend gửi frame đến API nhận diện.
  3. Backend chuyển ảnh sang AI service để detect khuôn mặt.
  4. Nếu phát hiện mặt, AI service trích xuất embedding và query Qdrant.
  5. Hệ thống nhận top-k vector gần nhất và tính khoảng cách hoặc similarity.
  6. Nếu điểm số vượt threshold thì xác định người dùng hợp lệ, ngược lại gán là unknown.
  7. Backend lưu log, snapshot và trả về kết quả cho frontend user.
  8. Frontend hiển thị tên, confidence, thời gian và trạng thái access.

Gợi ý kỹ thuật

  • Face detection: dùng SCRFD, RetinaFace hoặc detector đi kèm trong InsightFace.
  • Face recognition: dùng ArcFace hoặc Facenet. Nếu cần làm nhanh thì dùng DeepFace để bọc pipeline.
  • Vector search: lưu embedding vào Qdrant, mỗi vector gắn person_id và metadata.
  • Decision rule: nếu similarity vượt threshold thì kết luận cùng người, ngược lại là unknown.

Điểm nên nói khi bảo vệ

  • Vì sao cần nhiều ảnh mẫu trên một người.
  • Vì sao phải có bước align để tăng độ ổn định.
  • Threshold là tham số vận hành, không nên hard-code cứng.
  • Hệ thống có thể mở rộng sang nhiều camera và nhiều người dùng hơn.
Không cần tự huấn luyện model mới; dùng model có sẵn hợp lý và đóng gói thành hệ thống vẫn phù hợp đề. Giải thích AI
Thiết kế dữ liệu

Lưu gì ở PostgreSQL, MinIO và Qdrant

Ba lớp lưu trữ phải có lý do tồn tại riêng. Trình bày rõ phần này giúp nhóm ghi điểm ở tiêu chí đáp ứng yêu cầu hệ thống.

09 / 15

PostgreSQL

  • admins: tài khoản đăng nhập quản trị.
  • persons: hồ sơ người được phép truy cập.
  • face_images: metadata ảnh mẫu, trạng thái xử lý.
  • recognition_logs: thời gian, kết quả, confidence.
  • jobs: trạng thái embedding và re-index.
  • settings: threshold, cấu hình camera.

MinIO

  • Lưu ảnh gốc mà admin upload.
  • Lưu snapshot frame khi hệ thống nhận diện hoặc từ chối.
  • Lưu dữ liệu theo bucket tách riêng: faces, snapshots, exports.
  • Không nhét file ảnh lớn vào database.

Qdrant

  • Mỗi ảnh mẫu sinh ra một vector embedding.
  • Payload có thể gồm: person_id, image_id, role, created_at.
  • Search top-k vector gần nhất để xác định danh tính.
  • Dễ mở rộng khi số người và số ảnh tăng lên.

Ví dụ bảng log

id, person_id, camera_id, result, confidence, snapshot_url, created_at

Ví dụ payload vector

{ person_id, image_id, source: "admin_upload" }

Ví dụ object key

faces/person_12/img_03.jpg

snapshots/2026-05-20/cam_1_080214.jpg

Thành viên 2 - Backend API + PostgreSQL + MinIO Thiết kế schema DB, auth admin, CRUD persons, upload ảnh, log nhận diện, gọi AI service, trigger queue, quản lý metadata ảnh trong MinIO.
Thành viên 3 - Frontend user Làm giao diện camera realtime, hiển thị tên, confidence, trạng thái access, danh sách log gần nhất và các trạng thái lỗi hoặc loading.
Thành viên 4 - Frontend admin + Docker + nginx + README Làm màn hình quản trị người dùng, upload ảnh, theo dõi job, xem log; đồng thời viết Dockerfile, docker-compose, nginx.conf, .env.example và tài liệu chạy.

Những gì phải chốt chung từ sớm

  • Format response từ AI service sang backend.
  • Schema dữ liệu persons, face_images, logs.
  • API contract cho frontend user và frontend admin.
  • Tên service, port, env var để viết Docker Compose.

Điểm chốt nên nói

  • Đề tài không chỉ nhận diện khuôn mặt, mà là một hệ thống AI full-stack hoàn chỉnh.
  • Luồng quản trị và sinh embedding nền chứng minh nhóm đáp ứng đúng yêu cầu về storage, vector DB và queue.
  • Docker Compose giúp giảng viên tái hiện hệ thống bằng một lệnh duy nhất.
  • Thiết kế service rõ ràng giúp dự án dễ mở rộng thành nhiều camera hoặc nhiều điểm kiểm soát trong tương lai.

Kịch bản demo 3 phút

  1. Đăng nhập admin, mở danh sách người dùng và ảnh mẫu.
  2. Cho xem trạng thái job embedding đã hoàn tất.
  3. Chuyển sang màn hình user và quét khuôn mặt đã đăng ký.
  4. Cho xem trường hợp unknown để chứng minh threshold hoạt động.
  5. Mở trang log để xác nhận snapshot và lịch sử sự kiện đã được lưu.

Checklist trước khi nộp

  • Repo private, thêm đủ thành viên và giảng viên.
  • docker-compose up -d chạy được từ đầu.
  • Không commit secret, có .env.example.
  • README có sơ đồ kiến trúc, hướng dẫn chạy, hướng dẫn test.
  • Quay video hoặc chuẩn bị demo trực tiếp với ít nhất 2 tình huống.
